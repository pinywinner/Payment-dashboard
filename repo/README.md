# דשבורד תשלומים — מדריך הקמה

## מה יש כאן

```
/shared/          העיצוב והלוגיקה. משותף לכל הלקוחות. עורכים פעם אחת.
/asraf/           לקוח. קובץ אחד בן 30 שורות.
/_template/       תבנית ליצירת לקוח חדש.
```

הכלל: **הנתונים בגוגל שיטס, הקוד בגיטהאב.** עדכון נתונים לא דורש נגיעה בקוד.

## שלב 2 — להעלות לגיטהאב

1. צור repo **פרטי** חדש.
2. גרור לתוכו את כל התיקיות מכאן.
3. פתח את `asraf/index.html` והדבק את ה־sheetId במקום `PASTE_THE_SHEET_ID_HERE`.

---

## שלב 3 — לחבר ל־Cloudflare Pages

1. חשבון חינמי ב־Cloudflare ← Workers & Pages ← Create ← Pages ← Connect to Git.
2. בחר את ה־repo. **Build command:** להשאיר ריק. **Output directory:** `/`.
3. Save and Deploy.

מהרגע הזה, כל שינוי שתדחוף לגיטהאב מתפרסם לבד תוך כדקה.

---

## שלב 4 — לחבר דומיין

1. קנה דומיין (Cloudflare Registrar — אותו חשבון, פחות דברים שיישברו).
2. בפרויקט ה־Pages ← Custom domains ← Set up a domain.

הקישור של הלקוח: `yoursite.com/asraf`

---

## שלב 5 — להגן על הקישור

הכתובת ניתנת לניחוש, והנתונים רגישים. ב־Cloudflare:
Zero Trust ← Access ← Applications ← Add an application ← Self-hosted.
Path: `asraf`, ומדיניות של רשימת אימיילים מורשים. חינם עד 50 משתמשים.

הלקוח מקבל קוד חד־פעמי במייל בכניסה הראשונה. בלי סיסמה לזכור.

---

## שגרת העבודה

**עדכון נתונים** — פותח את השיטס, מוסיף שורה ב־Payments, מעדכן את `Last updated` ב־Config. זהו. הלקוח מרענן ורואה.

**לקוח חדש** —
1. שכפל את הגיליון, מלא נתונים.
2. העתק את `_template/` לתיקייה בשם הלקוח.
3. החלף בקובץ: `CLIENT NAME`, `CLIENT-SLUG`, `YOURSITE.com`, וה־sheetId.
4. דחוף. הקישור חי.

**שינוי עיצוב** — עורך קובץ אחד ב־`/shared/`, דוחף, וזה משתנה אצל כל הלקוחות.

---

## מה קורה כשמשהו לא מסתדר

הדשבורד בודק את עצמו לפני שהוא מציג. אם שולם + מתוכנן לא שווה לאומדן הכולל, או אם ספק קיבל יותר מהתקציב שלו, מופיע באנר שמפרט בדיוק מה לא מסתדר.

ברירת המחדל: להציג בכל זאת עם אזהרה. אם תעדיף שהדשבורד פשוט יסרב להיטען — שנה ב־`index.html` של הלקוח:

```js
blockOnMismatch: true
```

**התאריכים** — אם השיטס מציג `05/04/2026` בפורמט אמריקאי, שנה ל־`dateFormat: 'MDY'`. טעות כאן מזיזה תשלומים בין חודשים בגרף בלי להתריע.

## בדיקה לפני שזה מגיע ללקוח

**שולם + נותר = אומדן כולל.** אם כן — הכל התחבר נכון.

# נספח — גיליון אחד לכל הלקוחות

במקום גיליון נפרד לכל לקוח, אפשר גיליון־אב אחד. מוסיפים עמודת `Client` בתחילת כל טאב:

| Client | Date | Vendor | Scope | Paid |
|---|---|---|---|---|
| asraf | 13/11/2025 | Hershy Klein | Construction Works | 590000 |
| levi | 02/01/2026 | ... | ... | ... |

ובטאב `Config`, שלוש עמודות: `Client | Key | Value`.

ומוסיפים טאב חמישי, `Clients`:

| Client | Token | Active |
|---|---|---|
| asraf | k7f2m9x4qp | yes |
| levi | b3n8v1c6rt | yes |

הטוקן הוא סתם מחרוזת אקראית שאתה ממציא. הוא מה שמזהה את הלקוח.
`Active` על `no` מנתק גישה ללקוח מיד, בלי לגעת בקוד.

## למה צריך את ה־Apps Script

בשיטה של גיליון מפורסם, הדפדפן של הלקוח מוריד את **כל** הקובץ ואז מסנן.
כלומר לקוח א' יכול לראות בקלות את הנתונים של לקוח ב'. הסינון קוסמטי בלבד.

ה־Apps Script פותר את זה: הגיליון נשאר **פרטי לחלוטין** (לא משותף, לא מפורסם),
והסקריפט הוא הדלת היחידה החוצה. הוא מקבל טוקן, מסנן בצד של גוגל, ומחזיר רק
את השורות של אותו לקוח. מה שלא שייך ללקוח פשוט לא עוזב את הגיליון.

## התקנה

1. בגיליון: Extensions ← Apps Script.
2. מחק את מה שיש, הדבק את `apps-script/Code.gs`, שמור.
3. Deploy ← New deployment ← Web app.
   - Execute as: **Me**
   - Who has access: **Anyone**
4. אשר את ההרשאות. העתק את הכתובת שמסתיימת ב־`/exec`.
5. ב־`index.html` של כל לקוח: מחק את `sheetId`, ובטל את ההערה משתי השורות
   `apiUrl` ו־`token`.

הגיליון עצמו — אל תשתף אותו עם אף אחד. זה כל הרעיון.

## מתי זה שווה את זה

**גיליון לכל לקוח** — פשוט יותר, אין קוד להתקין, וההפרדה מובנית.
עדיף עד בערך חמישה לקוחות.

**גיליון־אב** — עדכון אחד במקום לפתוח חמישה קבצים, ואפשר לראות תמונה
כוללת של כל הפרויקטים. המחיר: תלות בסקריפט, ואם הוא נשבר — כולם נופלים
יחד. אחרי כל שינוי בסקריפט צריך Deploy מחדש (**Manage deployments ←
עריכה ← New version**), אחרת השינוי לא נכנס לתוקף.
